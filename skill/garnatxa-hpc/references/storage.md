# Storage and data movement on Garnatxa

## Filesystem layout

| Path | Purpose |
|------|---------|
| `/home/<USER>` | Personal directory. Created on first login. Visible on the login node and all compute nodes. For your software, configs, small data. |
| `/storage/<GROUP>` | Per-group shared area for processing large files. Code, inputs and outputs that have to be shared between group members live here. |
| `/storage/<GROUP>/home_members/<USER>` | Per-user personal directory inside the group's allocation. |
| `/scr` | Scratch for files generated by batch jobs. Purged periodically — only use it as transient working space. Copy results to `/storage` or `/home` before deleting them. |
| `/tape2/<TAPE_CODE>/` | LTO-9 tape mount point (e.g. `/tape2/GRP0000L9/`). |

Primary filesystem is **Ceph Reef 18.2.1**. There is no `/lustre`.

**No backups.** `/home`, `/storage`, `/scr` — none are backed up. Once a file is
deleted, it's gone. Use the tape library (next section) for archival. Keep
off-cluster copies of anything truly irreplaceable.

## Quotas

Quotas are computed at the **group** level on `/storage/<group>`, monitored
daily. Key thresholds (M = total system capacity):

- **Soft limit**: 75% of M — alarm and warning.
- **Hard limit**: 80% of M — mandatory write restriction; filesystem may go
  read-only.
- Ceph self-maintenance needs ~20% headroom.

Groups contribute storage; effective usable share is:

- After Ceph 3× replication: net = raw / 3.
- Of that, 80% is usable by I2SysBio groups (70% for external).
- `Effective_size = Used − γ × Contributed`, where γ = 0.80 (I2SysBio) or
  0.70 (external).

When the soft limit is crossed, a dynamic per-group quota `Q` is applied
automatically until the cluster recovers headroom. Groups above `Q` must shed
data; if the hard limit is hit each group is held to `Q + γ · a_k`.

There is no documented per-user disk quota — restrictions are imposed on the
group folder.

## Check what's there before doing anything expensive

```bash
checkdiskspace          # safe alternative to `du` for big trees
du_                     # safe alternative to `du`
ls /tape2/<TAPE_CODE>   # list contents of a tape mount
tapecopy -l             # list tapes with capacity / free space / use%
```

Avoid `find`, `ls`, or `du` over directories holding > ~10 000 files — they
hammer the filesystem.

## Transferring files

The login host is `garnatxa.srv.cpd`. VPN first if outside the UV network.

### scp (use `-O` for compatibility)

```bash
# local -> Garnatxa
scp -O ./file.txt USER@garnatxa.srv.cpd:./my_garnatxa_dir/

# Garnatxa -> local
scp -O USER@garnatxa.srv.cpd:./my_garnatxa_dir/file.txt .

# recursive
scp -O -r USER@garnatxa.srv.cpd:/path/to/dir .
```

### rsync (recommended)

```bash
rsync --inplace --progress --partial --append -av ./local_dir \
      USER@garnatxa.srv.cpd:.
```

Resumes interrupted transfers and skips files that are already in sync.

### Windows

- **WinSCP** (SFTP/SCP GUI): <https://winscp.net/eng/download.php>.
- **WSL** (Windows Subsystem for Linux) to run the Linux commands above.

## Splitting big files

For when you want to parallelize across smaller chunks. Note: there is no
documented "split for tape" command — `tapecopy` handles multi-tape spillover
itself.

```bash
split --numeric-suffixes -l <LINES_PER_CHUNK> <INPUT> <OUTPUT_PREFIX>
```

Example — split a 14 010 000-line FASTQ into 1 000 000-line chunks:

```bash
split --numeric-suffixes -l 1000000 reads_00.fq subreads00_
ls subreads00_*
# subreads00_00  subreads00_01  ...  subreads00_14
```

## Tape archive (LTO-9)

Hardware: IBM TS4300, 2 drives, 40 slots × LTO-9 16 TB (~720 TB online).
Tape life ~30 years. Sequential-access media — **deleting a file does NOT
reclaim space**; only reformatting does.

### Requesting tape access

1. Open a ticket on <https://garnatxadoc.uv.es/support> under topic
   `Garnatxa HPC`. Each LTO-9 tape is paid by the group/collaboration
   (~€100 per tape).
2. Admins prepare the tape so only the owning group can access it. It appears
   as a path like `/tape2/GRP0000L9/`.
3. The tape stays mounted while you store/retrieve. Afterwards it's
   physically removed and stored.

### `tapecopy` — write to tape

You must `ssh merlot` first — tape operations don't run on the login node.

```bash
ssh merlot

tapecopy -h                          # help
tapecopy test/a.out                  # one file
tapecopy test/output_dir             # whole directory
tapecopy "test/out/*.sai"            # quote wildcards
tapecopy -l                          # list tapes (capacity, available, use%)
```

Each invocation submits a SLURM job and writes a `jobtape_<timestamp>.out`
log so you can disconnect and check later.

Behaviour:

- Picks the first tape with enough free space.
- If no tape fits the whole payload, nothing is copied — split first or buy
  more tape.
- If multiple tapes are mounted in `/tape2`, large payloads spill across
  them automatically.
- Files already on the tape are skipped.

### Restoring files from tape

Tape mounts are visible from the login node as `/tape2/<TAPE_CODE>/`, but
**only the owning group has read permission** — a plain `ls /tape2` returns
`Permission denied` for anyone outside the group that bought the tape. Once
you have access:

```bash
cp /tape2/XXX006L9/home/user/test/out/file.sai /home/user/test/out
```

Possible — but **not advisable** — to read directly inside a job (sequential
media; it slows everything down).

> Behind the scenes, `tapecopy` is a wrapper that calls the real binary
> `copyToTape` on `merlot`. The user-facing command is still `tapecopy` (and
> that's what the help banner exposes) — useful to know only if you're
> debugging a failure and see `copyToTape` in error output.

## Decision table

| What you want to do | Where to put / get it |
|---------------------|-----------------------|
| Configs, dotfiles, small custom scripts | `/home/<USER>` |
| Group code, refs, processed datasets | `/storage/<GROUP>/...` |
| Per-user data inside the group quota | `/storage/<GROUP>/home_members/<USER>` |
| Transient work during a batch job | `/scr` |
| Archive a finished project | tape (`ssh merlot`, `tapecopy`) |
| Bring data on/off the cluster | `rsync` (preferred) or `scp -O` |
| Transfer from Windows | WinSCP or WSL |
| Inspect tape contents | `ls /tape2/<TAPE_CODE>` |
| Find how much space is free on tapes | `tapecopy -l` |
